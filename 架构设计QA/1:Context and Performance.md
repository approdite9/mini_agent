Q1.1:大模型面对第一轮长窗口或多模态输入时，first token 会显著变慢。有什么快速/低成本/用户体验也不差的方案？从5-10秒稳定压缩到2秒。

A:应该主要是看排队时间 + prefill 时间 + 首 token decode，需要先测不同指标瓶颈进行相对方案调试。（简单概括）若无法解决可进行感知欺骗，先用小模型/规则给出概括/推测，随后流式传输，结构化输出场景,先把标题/框架流出来。让用户在长时间等待中提高体验感。


Q1.2:一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

A:先想清楚"压缩"到底压什么。200 轮对话里,信息的价值密度是极不均匀的:早期的寒暄、已经被推翻的方案、中间的试错,价值很低;但用户的关键偏好、已确认的事实、当前正在进行的任务状态、最近几轮,价值极高。所以压缩的本质不是均匀缩小,而是"按价值分层保留"。
我认为的模型是一个分层的 context 结构:
* 永久层(pinned facts):用户是谁、核心偏好、项目背景、已达成的关键决定。这些应该被抽取成结构化的、极短的事实条目,几乎不参与压缩,始终在 context 里。
* 摘要层(rolling summary):对早期对话做递归摘要——每当对话增长到阈值,就把最老的一段 K 轮压成一段摘要,追加到"摘要区",原始轮次移出 context。摘要是有损的,但保留了叙事连贯性。
* 原文层(recent verbatim):最近 N 轮保持原文,因为对话的流畅性极度依赖最近上下文的字面细节(指代、语气、刚说的具体数字)。这一层不能压。
压缩的时机和方式。 有两种方式:
* 递归摘要(recursive/rolling summarization):滑动窗口,老内容持续被折叠进摘要。优点是 context 稳定不爆;缺点是信息一旦进摘要就模糊了,多次递归摘要会累积失真(摘要的摘要),而且用户如果突然回头问一个很早的细节,可能已经丢了。
* 检索式(把历史存外部,按需召回):所有历史存向量库/KV,当前 context 只放摘要+最近轮,用户问到相关内容时再检索原文塞回来。优点是不真正丢信息;缺点是检索会引入延迟和"召回错/召回不全"的问题,而且检索进来的片段和当前对话拼接可能不连贯。
我倾向两者结合:滚动摘要保证叙事连贯 + 关键原文存外部索引。当摘要不足以回答、用户明显在追问历史细节时,触发检索把原文捞回来。
真实场景中，如何保证压缩后"仍然流畅"——这是真正的难点。 流畅性容易被破坏的地方:
* 指代断裂:摘要里把"那个方案"具体化了,但用户下一句还说"那个方案",模型对不上。解法:摘要时保留关键实体的显式命名,别过度抽象。
* 语气/风格漂移:摘要通常是干巴巴的第三人称,把它塞回去后模型可能突然变得公事公办。解法:把"对话风格/关系状态"也作为一条 pinned fact 显式记录("用户偏好轻松口语、已建立熟悉感"),让模型知道该用什么口吻。
* 压缩发生的"接缝":如果在某一轮突然做压缩,模型的回答可能出现前后不一致。解法:异步压缩——不在关键回合的关键路径上做,而是在用户思考的间隙、或对话稍冷时后台压缩,压缩结果下一轮才生效。
还有一个我认为容易被忽略的点:压缩要不要让用户可感知/可控? 有的产品做"记忆卡片"让用户看到 Agent 记住了什么、可以编辑删除。这既提升信任,也给了纠错入口(摘要错了用户能改)。代价是产品复杂度和打断感。
收敛。 
我的方案:三层结构(pinned facts / rolling summary / recent verbatim)+ 关键原文外部可检索 + 异步压缩避开关键路径 + 把"关系状态和风格"也当事实记录来维持口吻连贯。评估上,我会重点比较"压缩后模型答错历史事实的比例"和"指代解析失败率"这两个指标,而不是只看 token 数降了多少。
