Q2.1:和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？

A:"以前问过的问题"——这里有个关键的语义分叉,用户是想要"同样的答案",还是这半个月里情况变了、想要"更新的答案"? 这决定了召回的正确行为。
场景 A:用户忘了问过,重复问。理想行为是:Agent 认出"你之前问过类似的,我当时说的是 X",然后要么直接复用、要么补充"不过这半个月有些变化"。这体现了"熟悉感"和记忆的价值。
场景 B:用户明知问过,但想要更新。这时死板复用旧答案反而糟糕。
所以合理的召回不只是"找到旧问答",还要判断复用 vs 更新。
召回机制本身怎么设计。 单纯语义向量检索通常是:
* 只按语义相似度召回不够。同样一句"帮我看看这个方案",半个月前和现在指的可能是完全不同的方案。需要把时间、会话、话题实体作为额外的召回维度,而不是纯 embedding 相似度。混合检索:向量 + 关键词 + 时间衰减 + 实体匹配。
* 召回粒度:是召回"那一轮原文",还是召回"当时抽取的结论/事实"?我倾向分两级——先召回结构化的事实/结论(短、准),需要细节时再回溯原始对话。
* query 改写:用户现在的问法和当时可能不一样。用当前对话上下文把 query 补全(把"这个"解析成具体实体)再去检索,召回率会高很多。
"更合理"我理解的层次,从低到高:
1. 能召回到(基础的向量检索就能做,但容易召回噪声)。
2. 召回准(混合检索 + 时间/实体约束,减少张冠李戴)。
3. 召回后会判断(认出这是重复问题,决定复用还是更新,并显式告诉用户"你之前问过")。
4. 主动性(甚至不用等召回——如果 Agent 在写答案时"想起"相关旧事,自然带出来,像人一样)。
第 3、4 层才是把"记忆"做出体验的地方。技术上,第 3 层可以在生成前加一个判断步骤:检索到旧问答后,让模型判断"当前 query 与旧记录是否等价、旧答案是否仍然有效",据此选择复用/更新/融合。
召回不能太慢也不能太吵。每轮都做重检索会拖慢 TTFT(模块一有提到),而且召回错的记忆比没有记忆更伤体验(自信地记错)。所以我会设召回触发条件——不是每轮都全量检索,而是检测到"这可能涉及历史"(比如指代模糊、用户提到过去、问题像是查询类)才触发,平时走轻量的最近上下文。

Q2.2:你理解的 Agent memory 经典框架是什么？它的发展趋势是什么，最头部的玩家在怎么做？

A:经典框架（常见八股中的划分）按时间跨度:短期/长期记忆，本质是context和跨session持久化。
按内容类型，语义记忆/情景记忆/程序记忆，主要是借用认知科学的类比(很多框架都这么分)。
按读写机制,一个记忆系统本质是四个操作的组合:抽取(什么该被记)→ 存储(怎么组织)→ 召回(怎么找回)→ 更新/遗忘(旧记忆冲突、过时怎么办)。
我认为更新和遗忘是最难也最被低估的一环——大多数系统只做了"加",不做"改"和"忘",结果记忆库里堆满过时和矛盾的条目,越用越差。

代表性的做法/流派(概念层面,不绑定具体产品名以免过时):
* 摘要式:把历史滚动摘要成长期记忆(简单,但有损、难更新)。
* 向量检索式(RAG-for-memory):每条记忆存成 embedding,按需检索。是目前最主流的骨架,但纯向量有前面说的语义漂移问题。
* 结构化/知识图谱式:把记忆抽成实体-关系三元组存图,召回时能做多跳推理、能显式处理"更新"(改一条边)。表达力强、可解释、好更新,但抽取成本高、维护复杂,而且不是所有记忆都适合图结构。
* 分层记忆(受操作系统启发):把有限的 context 当"内存",外部存储当"磁盘",让 Agent 自己决定什么调入调出——这是把 memory 管理本身交给模型来做的思路,比较有代表性。
* 自反思/自演化记忆:Agent 定期回顾自己的经历,提炼出更高层的教训/程序性知识存起来,而不只是存原始事件。

发展趋势:
1. 从"存原文/存摘要"走向"存结构化的、可更新的、有类型的记忆"——尤其是 semantic + procedural 的显式建模。
2. 从"被动检索"走向"主动记忆管理"——Agent 自己决定记什么、忘什么、什么时候把记忆调进 context,而不是外部规则硬编码。
3. 遗忘/衰减/冲突消解会成为重点。人类记忆的核心特征是会忘、会随时间重构,当前系统在这块几乎是空白。
4. 记忆与个性化、与"关系"融合——memory 不只是事实库,还承载 Agent 和用户之间关系状态的连续性。
5. 评估标准化滞后:目前很难客观评价一个记忆系统好不好,这本身会成为一个需要解决的问题。

头部玩家:开源有若干专门做 agent memory 的库/框架,也是分别使用向量检索、知识图谱、分层管理这几条路线，或混合使用。
