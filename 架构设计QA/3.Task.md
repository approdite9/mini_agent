Q3.1:对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

A:长程任务里,原始目标在 context 里离当前位置越来越远,被大量中间步骤、工具输出、试错淹没。注意力被稀释,加上如果中途做了 context 压缩(模块一),目标可能被摘要糊掉甚至丢掉。表现为:偏离原始意图、陷入局部子任务出不来、或反复绕圈。

解法一:目标常驻(goal pinning)。把原始目标 + 成功标准放在一个永不被压缩、永远在 context 里的固定位置(通常是 system 区或每轮 prompt 顶部)。最简单有效。
* 优点:实现简单,直接对抗"离得远被稀释"。
* 缺点:只对抗"忘",不对抗"跑偏"。目标还在,但模型仍可能在执行中偏离;而且目标如果本身是模糊的,pin 住模糊目标没用。

解法二:显式计划 + 进度追踪(plan & todo list)。任务开始时让模型生成一个结构化的 plan / todo,执行中不断更新状态(done / doing / todo),每步都对照 plan。这就是很多 coding agent 里的 todo 机制。
* 优点:不仅记得目标,还记得"到哪了、还剩啥",天然对抗跑偏和绕圈;plan 可见还能让用户干预。
* 缺点:plan 可能一开始就错(错误的分解),之后一直照错的走;plan 维护本身消耗 token 和步骤;动态性强的任务里僵化的 plan 反而是负担。

解法三:分层/递归分解(hierarchical / manager-worker)。上层 Agent 只管目标和分解,把子任务派给下层 Agent 执行,下层的中间过程不污染上层 context。上层始终在"目标层"思考。
* 优点:目标层 context 干净,不会被执行细节淹没;可扩展到很复杂的任务。
* 缺点:架构复杂;层间信息传递会丢失/失真;子 Agent 失败如何回传和重规划很麻烦;整体更贵更慢。

解法四:周期性自我校准(reflection / re-grounding)。每隔 N 步,强制插入一次"我现在的目标是什么?我偏离了吗?下一步该做什么?"的反思。
* 优点:主动纠偏,不只是被动记住;能发现绕圈。
* 缺点:反思步骤增加成本和延迟;反思本身也可能出错(自我欺骗式地判断"我没跑偏");频率难调,太频繁啰嗦,太稀疏没用。

解法五:外部化状态(state 存在 context 之外)。目标、计划、进度都存在一个外部结构化 store 里,每轮从 store 读取最关键的部分注入,而不是依赖 context 累积。本质是把"记忆"从模型的注意力里搬到确定性的程序里。
* 优点:确定性强,不受 context 长度和压缩影响;可审计。
* 缺点:需要设计好 state schema;什么时候读写、注入多少,仍需策略;把灵活性交给了外部程序,可能不够自适应。

真实开发场景中这几种不是互斥的,一般都是根据场景混合使用:外部化的目标+计划状态(解法五)作为骨架,todo 追踪(解法二)作为进度,周期性 re-grounding(解法四)作为纠偏,复杂任务再上分层(解法三)。单靠 pinning 太弱,单靠反思太贵且不稳。核心洞察是:别指望模型靠"记性"记住目标,要用确定性的程序结构把目标"托住",模型只负责在每一步做局部推理。 长程可靠性来自架构,不来自更长的 context。
