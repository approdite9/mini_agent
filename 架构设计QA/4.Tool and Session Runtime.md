
Q4.1:Agent 工具有同步和异步两类。异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？

A:Claude Code有类似这样针对此场景的设计。
核心矛盾主要是:不能让用户干等,但结果又重要、不能丢。 把它拆成三个子问题:提交、追踪、回填。

提交。Agent 调用异步工具时,工具立刻返回一个 task_id(而不是最终结果),表示"已受理,在跑了"。此刻 Agent 应该给用户一个明确反馈:"我已经开始生成视频了,好了会告诉你,你可以先聊别的。" 这一步把"重要的东西在后台跑"这件事对用户透明化——不透明的等待才是坏体验。

追踪。task_id 对应的任务状态存在一个持久的 task store 里(pending / running / done / failed),独立于对话 session 的生命周期——因为用户可能关掉页面、几小时后回来,任务不能因为 session 结束就丢。追踪有两种拿结果的方式:
* 轮询(polling):runtime 定期查任务状态。简单、通用,但有延迟和无效查询开销。
* 回调/事件(webhook/callback):任务完成时主动推一个事件给 runtime。及时、高效,但需要工具方支持回调,且要处理回调丢失/乱序/重复。现实里常常是混合:优先回调,同时用低频轮询兜底(防回调丢失)。

回填。任务完成事件到达时,结果要回到对话里。这里是设计的关键分叉:
* 结果到达时,用户可能正在和 Agent 聊别的(session busy),或者用户已经离开。
* 结果不能粗暴打断当前对话流(想象用户正问着别的,突然蹦出十分钟前那个任务的输出,很错乱)。

我倾向的设计:异步结果到达后,不直接插入对话流,而是先进一个"待处理结果队列",然后:
* 如果用户在线且空闲:Agent 主动播报"你之前让我生成的视频好了 👉",附上结果,并给出接续入口。
* 如果用户在忙(正在别的 turn 里):等当前 turn 结束,或用一个非阻塞的提示条(角标/通知条)告知,让用户自己决定什么时候查看,而不是打断。
* 如果用户离线:发推送/站内通知,结果留存,回来能看到。
几个必须处理的边界:
* 失败:异步任务失败了怎么通知、要不要自动重试、重试几次、最终失败怎么优雅告诉用户。
* 超时:跑太久了要有上限和状态反馈("还在跑,已经 5 分钟了")。
* 关联:结果回填时用户可能已经聊到很后面,Agent 要能说清"这是你 10 分钟前那个请求的结果",提供上下文锚点,否则用户懵。
* 多个异步任务并发:多个 task_id 要能各自追踪、各自回填、不串味。
收敛。 
即异步工具 = 提交即返回 task_id + 独立于 session 的持久任务状态 + 完成事件驱动 + 结果进队列而非直插对话 + 分"在线空闲/在线忙/离线"三种触达策略 + 失败超时兜底。 关键设计原则是任务生命周期和会话生命周期解耦,以及结果回填要尊重用户当前的注意力状态,不粗暴打断。
