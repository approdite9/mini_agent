
Q5.1:Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

A:
OpenAI-compatible function calling 范式。 模型在生成中吐出一个结构化的 tool_call(函数名 + JSON 参数),运行时解析、执行、把结果作为 tool 消息塞回对话,模型接着生成。特征是:工具调用是协议层的结构化字段(和文本内容分离的 tool_calls 结构),参数是 JSON,由 API 契约约束。国内很多模型做 OpenAI 兼容,基本沿用这套。
* 优点:标准化、生态兼容好(能直接接入所有为 OpenAI 格式写的工具/框架);参数结构化、机器可解析、易校验;与现有 API 基础设施契合。
* 缺点:JSON 作为动作表达不够灵活/表达力有限(嵌套复杂逻辑、组合调用、控制流很别扭);模型要严格产出合法 JSON,格式错误/参数幻觉是常见故障点;多步/并行工具编排在纯 function calling 下比较笨重。

Claude Code 的方式——依靠大模型和自身使用体验。 我的理解是,Claude 系模型的工具使用底层也是结构化的 tool_use 块,但在 Claude Code 这类 agentic coding 产品里,工具的组织和输出处理有一些不同取向:工具输出(尤其是文件、命令执行结果)可能被更结构化地管理和回注,强调 agentic loop 里工具结果的高效流转,以及和"模型自己规划、连续多步调用工具"的配合。也有一种流派是用代码作为动作(让模型写代码来调用工具,而非填 JSON),表达力和组合性更强。目前不能确定 Claude Code 当前具体是哪种实现细节。
* 优点(code-as-action 思路):表达力强,一段代码能表达循环、条件、多工具组合,一步顶 JSON 的多步;对复杂 agentic 任务更自然;工具输出能在代码环境里被程序化处理。
* 缺点:需要一个代码执行沙箱,安全和隔离成本高;模型写错代码的故障模式比填错 JSON 更难兜;标准化和生态兼容不如 function calling。
